Confirming Native Function Calls with bpftrace

A personal note on using Linux uprobes and stack traces to prove a native symbol is actually executed.

Problem: a Python-level path says a custom CUDA or C++ extension should run, but I want kernel-level proof that a particular native function is entered. Sampling tools such as py-spy can miss short calls; a userspace probe on the native symbol counts every function entry.

Recipe

First, find the symbol exported by the extension shared object. C++ names may be mangled, so keep the exact symbol that appears in the binary.

readelf -Ws <extension.so> | grep dualkv_varlen_fwd

# Alternative, with demangling for readability:
nm -D --defined-only <extension.so> | c++filt | grep dualkv_varlen_fwd

Attach a bpftrace uprobe to the symbol and count calls. This fires on every entry to the function.

sudo bpftrace -e '
uprobe:/path/to/flash_attn_2_cuda.so:_ZN5Flash21mha_dualkv_varlen_fwdE... {
  @dualkv_fwd = count();
}

interval:s:20 {
  print(@dualkv_fwd);
}'

To confirm the calling path, aggregate user-space stack traces at the probe point.

sudo bpftrace -e '
uprobe:/path/to/flash_attn_2_cuda.so:_ZN5Flash21mha_dualkv_varlen_fwdE... {
  @[ustack()] = count();
}

interval:s:20 {
  print(@);
  clear(@);
}'

Interpretation

If the counter grows while the workload runs, the native symbol is being executed. In my DualKV check, the uprobe count climbed to 12,914 calls over roughly 60 seconds, which was stronger evidence than config checks or sampling traces.

The stack-trace form answers the follow-up question: not only did the function run, but which user-space path reached it. This is the reusable trick when a Python binding, dispatcher, or CUDA extension makes the actual execution path hard to see.

Fallback

If bpftrace is unavailable, perf probe can attach to the same shared-object symbol.

sudo perf probe -x <extension.so> dualkv_varlen_fwd
sudo perf stat -e 'probe_*:dualkv_varlen_fwd' -p <pid>

Gotchas

Use the mangled symbol when needed. The process must have the shared object mapped, and probing usually needs sudo or CAP_BPF. Stack quality depends on frame pointers, unwind information, and symbol visibility. Inlined functions are not good probe targets.
