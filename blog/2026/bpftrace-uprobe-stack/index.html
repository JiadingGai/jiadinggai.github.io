Confirming Native Function Calls with bpftrace

Using Linux uprobes and stack traces to prove a native symbol is actually executed.

Problem: a Python-level path says a custom CUDA or C++ extension should run, but I want kernel-level proof that a particular native function is entered. Sampling tools such as py-spy can miss short calls; a userspace probe on the native symbol counts every function entry.

Here, dualkv_varlen_fwd is DualKV's variable-length forward FlashAttention path. DualKV shares a prompt across multiple rollout responses; tracing its compiled C++/CUDA symbol proves that the workload is actually dispatching the optimized path.

bpftrace Primer

bpftrace compiles a short tracing script to eBPF. This note uses four concepts:

  • uprobe:library:symbol fires when a process enters a user-space function in a shared object.
  • -p PID limits that uprobe to the target Linux process.
  • @map = count() aggregates probe hits without printing every event.
  • ustack() makes a user-space stack trace the map key. interval:s:20 prints it, and clear(@) starts the next reporting window.

Recipe

First, find the native symbol exported by the extension shared object. The Python entry point flash_attn_dualkv_varlen_func calls flash_attn_2_cuda.dualkv_varlen_fwd, which is bound to Flash::mha_dualkv_varlen_fwd. Get the exact mangled name:

readelf -Ws <extension.so> | grep dualkv_varlen_fwd
# Example result: _ZN5Flash21mha_dualkv_varlen_fwdE...

# Alternative, with demangling for readability:
nm -D --defined-only <extension.so> | c++filt | grep dualkv_varlen_fwd

Attach a bpftrace uprobe to the symbol and count calls. This userspace probe requires sudo or CAP_BPF and fires on every function entry.

PID=<target-pid>
sudo bpftrace -p "$PID" -e '
uprobe:/path/to/flash_attn_2_cuda.so:_ZN5Flash21mha_dualkv_varlen_fwdE... {
  @dualkv_fwd = count();
}

interval:s:20 {
  print(@dualkv_fwd);
}'

To confirm the calling path, aggregate user-space stack traces at the probe point.

PID=<target-pid>
sudo bpftrace -p "$PID" -e '
uprobe:/path/to/flash_attn_2_cuda.so:_ZN5Flash21mha_dualkv_varlen_fwdE... {
  @[ustack()] = count();
}

interval:s:20 {
  print(@);
  clear(@);
}'

Interpretation

For this DualKV check, the correct eBPF uprobe on the native Flash::mha_dualkv_varlen_fwd symbol reported 12,914 calls over roughly 60 seconds. That is direct evidence that this workload dispatched the optimized path, rather than merely being configured for it; the exact count will vary with the workload.

The stack-trace form answers the follow-up question: not only did the function run, but which user-space path reached it. This is the reusable trick when a Python binding, dispatcher, or CUDA extension makes the actual execution path hard to see.

Fallback

If bpftrace is unavailable, perf probe can attach to the same shared-object symbol.

sudo perf probe -x <extension.so> dualkv_varlen_fwd
sudo perf stat -e 'probe_*:dualkv_varlen_fwd' -p <pid>

Gotchas

Use the mangled symbol when needed. The process must have the shared object mapped, and probing usually needs sudo or CAP_BPF. Stack quality depends on frame pointers, unwind information, and symbol visibility. Inlined functions are not good probe targets.
